The problem

Model knowledge has a habit of escaping containment.

Most model knowledge does not live in one system. It leaks across Excel files, tickets, validation reports, migration notes, Slack decisions, and the heads of tired consultants. Martenweave gives that knowledge a spine.

  • Excel mappings scattered across folders and local drives.
  • Model decisions buried in tickets, comments, and meeting notes.
  • Validation reports that are not linked back to model rules.
  • Project knowledge that evaporates after handover.
  • Unclear field ownership and mystery mappings with no evidence trail.
  • SAP migration teams arguing over which file is true.
  • AI stuck summarizing fragments because the model layer is not structured.

What it does

A canonical model layer from scattered project artifacts.

Martenweave treats model truth as structured, reviewable files. Generated indexes, reports, search documents, and lineage views are rebuilt from that truth.

Canonical model files

Markdown and YAML frontmatter objects become the durable source of truth.

Deterministic validation

Objects, IDs, types, SAP context, and references are checked before indexing.

Object/reference consistency

Broken links, duplicate IDs, and wrong reference types are surfaced early.

Generated model index

SQLite, search JSONL, and lineage outputs are disposable and rebuildable.

Dataset/model gap detection

Compare sample datasets with the model before mock loads become meeting fuel.

Impact analysis

Trace field and attribute consequences across mappings, usage, and lineage.

Evidence and lineage tracking

Keep decisions, validation findings, fields, and mappings connected.

Patch proposal workflow

AI drafts reviewable changes instead of quietly rewriting model truth.

Human approval gates

Approved changes become change requests. The model does not mutate in the dark.

GitHub issue integration

Use issues for review, change tracking, and scenario feedback where it fits.

AI-ready model context

Give AI structured context it can reason over without pretending to be truth.

Use cases

Built for teams who need traceability before the next escalation.

SAP migration model registry

Problem: Migration scope, fields, and rules drift across files.

Helps: Canonical objects connect domains, contexts, fields, and mappings.

Value: Fewer truth disputes during mock loads and cutover planning.

MDM rule and ownership layer

Problem: Ownership is known socially, not operationally.

Helps: Rules, attributes, and decisions become referenceable objects.

Value: Governance teams can discuss changes with evidence.

Customer / Business Partner mapping control

Problem: Customer field behavior differs by context and system.

Helps: Attribute usage and SAP context rules keep field meaning explicit.

Value: Better mapping control without spreadsheet archaeology.

Supplier / Vendor model control

Problem: Supplier and vendor terms blur across source and target models.

Helps: Model objects separate business meaning from physical fields.

Value: Cleaner handoffs between process, data, and migration teams.

Dataset gap detection before mock load

Problem: Missing fields appear late, usually with calendar drama.

Helps: Compare profiled datasets with expected model endpoints.

Value: Earlier remediation and less validation ping-pong.

Validation report triage

Problem: Reports show failures without model context.

Helps: Findings link back to fields, rules, mappings, and decisions.

Value: Triage becomes evidence-led, not inbox-led.

AMS support knowledge continuity

Problem: Support inherits decisions but not the reasoning.

Helps: Canonical model history keeps context available after go-live.

Value: Less tribal knowledge loss between delivery and support.

Data governance impact analysis

Problem: Small field changes can affect reporting, load, and ownership.

Helps: Impact traversal shows connected objects and lineage.

Value: Better change decisions before downstream teams discover them.

AI-assisted model maintenance

Problem: AI can help, but unchecked changes are not governance.

Helps: AI proposes patches that validators and humans review.

Value: Faster maintenance without surrendering control.

Why not just...

Excel, Confluence, tickets, and ChatGPT still have jobs. This is a different job.

Excel

Stores fragments. Useful fragments, often named like a cry for help.

Confluence

Stores explanations. Helpful, but rarely validates references or field context.

Tickets

Store disputes, approvals, and archaeology. They are not a model layer.

ChatGPT

Can summarize and draft. It should not be asked to quietly redefine truth.

Martenweave

Stores validated model truth and makes it operational through checks, indexes, reports, lineage, and reviewable changes.

AI proposes.

Validators verify.

Humans approve.

AI philosophy

AI is useful. Silent mutation is not.

Martenweave does not ask AI to be the source of truth. That would be brave in the same way deploying directly to production on Friday is brave. AI proposes changes. Validators check them. Humans approve them.

  • Suggest patches from notes, reports, and issue context.
  • Extract evidence and draft issues for review.
  • Explain likely impact from structured model context.
  • Keep deterministic validators as the gate.
  • Prevent silent model mutation.

Architecture

Backend-first because model truth should survive the interface fashion cycle.

The canonical model files are the truth. Generated indexes are acceleration. Reports are evidence. Proposals are reviewable. Nothing important changes silently.

Source

Canonical files

Markdown and YAML objects in a repository.

Gate

Deterministic validation

IDs, types, references, and SAP context rules.

Acceleration

Disposable indexes

SQLite, JSONL search documents, and lineage exports.

Value

Reports and workflows

Health, gaps, impact, audit, issues, and review queues.

CLI/API first. UI later, only where it helps. GitHub issues can support review and change workflows without becoming the model database.
