Why teams need it

Model knowledge gets scattered before governance can use it.

Before

  • Field definitions live in spreadsheets.
  • Decisions are buried in tickets.
  • Validation findings drift away from rules.
  • Dataset gaps appear late.
  • AI has no trusted context.

After

  • Model objects are structured.
  • Rules and fields are linked.
  • References are validated.
  • Gaps and impact are visible.
  • AI proposes changes, humans approve.

What Martenweave does

A lightweight control layer for model work.

import

Capture model knowledge

Bring in spreadsheets, datasets, notes, tickets, and validation evidence.

registry

Store canonical objects

Represent domains, attributes, field endpoints, mappings, rules, issues, and decisions.

validation

Validate references and rules

Check IDs, object types, broken links, ownership, and SAP context rules.

index

Build searchable indexes

Generate SQLite and JSONL outputs from validated canonical files.

gaps

Detect dataset and model gaps

Compare expected model endpoints with datasets before mock-load pressure rises.

impact

Trace impact and export context

Follow relationships across fields, rules, mappings, reports, and proposals.

Use cases

SAP migration model control

Problem: Fields and rules drift across workbooks.

Helps: Context, endpoints, mappings, and decisions stay linked.

Value: Better mock-load and cutover readiness.

MDM rule and ownership layer

Problem: Ownership is known socially, not operationally.

Helps: Rules, owners, and decisions become referenceable.

Value: Governance discussions start from evidence.

Customer / BP mapping

Problem: Field behavior changes by context and system.

Helps: Attribute usage and SAP table fields stay explicit.

Value: Less spreadsheet archaeology.

Supplier / Vendor control

Problem: Supplier and vendor meanings blur across contexts.

Helps: Business meaning is separated from physical fields.

Value: Cleaner handoffs across process, data, and support teams.

Dataset gaps before mock load

Problem: Missing fields appear late.

Helps: Dataset profiles are compared with expected endpoints.

Value: Earlier remediation.

AI-ready governance context

Problem: AI summaries are risky without trusted structure.

Helps: AI drafts PatchProposals for review.

Value: Faster maintenance without silent mutation.

Open-source core

Available today as a Python package and CLI.

The core is local-first and file-backed. It can be embedded in CLI workflows, repositories, local automation, API processes, MCP servers, and future UI.

Available now

  • Python package and CLI: martenweave-core 0.4.1
  • Canonical Markdown/YAML model files
  • Deterministic validators
  • SQLite, search, and lineage indexes
  • Gap, health, scorecard, trace, and impact commands
  • AI PatchProposal workflow with human approval gates

Integration direction

  • Imports create profiles or reviewable PatchProposals.
  • Exports produce CSV, XLSX, schemas, reports, or context files.
  • GitHub publishing is for issue and change review, not hidden mutation.
  • Hosted workbench ideas are future-facing, not available as SaaS today.
python -m pip install martenweave-core
martenweave init ./my-model
martenweave validate --repo ./my-model
martenweave build-index --repo ./my-model --jsonl
martenweave impact FEP-S4-KNVV-KDGRP --repo ./my-model
